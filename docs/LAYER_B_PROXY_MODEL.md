# Layer B Proxy Model

Layer B remains proxy-only. It is designed to make controller-law contrasts legible under small, bundled mechanistic tasks. It does not expose latent adequate-family truth.

## Proxy construction layers

1. Task-authored metadata

- suite membership
- probe purpose
- target hypotheses
- primary proxy metrics
- family proxy hint

2. Automatically derived proxy features

- normalized family proxy key
- target-file count
- resolving-family count
- bundle suite and branch-profile count
- bundle-level compression and contamination settings

3. Runtime agent-state-derived proxies

- `represented_candidate_family_count_proxy`
- `recoverable_candidate_count_proxy`
- `compression_collision_proxy`
- `avoidable_retirement_proxy`
- `reset_helpfulness_proxy`
- `stale_continuation_proxy`
- `post_reset_resolution_improvement_proxy`

## What these proxies can and cannot do

They can:

- expose whether controller moves preserve or collapse candidate-family diversity
- expose whether alias-prone compression correlates with loss of recoverability
- expose whether reset appears to help on stale-continuation probe tasks

They cannot:

- recover world-exact adequate-family truth
- certify causal validity outside the authored task slice
- substitute for benchmark-scale external validation

## Validity notes

Each bundled task asset includes a `proxy_validity_note` with:

- what the proxy can capture
- what it cannot capture
- likely false positives
- likely false negatives

These notes should be read before interpreting Layer B controller differences.
